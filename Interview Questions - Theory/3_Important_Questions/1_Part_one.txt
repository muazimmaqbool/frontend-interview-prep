Question 1: Component lifecycle methods vs Hooks lifecycle (Mounting, Updating, Unmounting) ?
Answer:
    -> In class components, React provides lifecycle methods like componentDidMount, componentDidUpdate, and componentWillUnmount to 
       perform actions during different phases of a component's lifecycle.

    -> In functional components, these lifecycle methods are replaced by the useEffect hook. 
       By controlling the dependency array, useEffect can mimic mounting, updating, and unmounting behavior.

    | Lifecycle Phase | Class Component          | Functional Component (Hooks)               |
    | --------------- | ------------------------ | ------------------------------------------ |
    | Mounting        | `componentDidMount()`    | `useEffect(() => {}, [])`                  |
    | Updating        | `componentDidUpdate()`   | `useEffect(() => {}, [dependencies])`      |
    | Unmounting      | `componentWillUnmount()` | `useEffect(() => { return () => {} }, [])` |

Question 2: Explain React reconciliation and the Virtual DOM diffing algorithm ?
Answer:
    ->React Reconciliation is the process React uses to determine what has changed in the UI and update only the necessary parts of the 
      DOM instead of re-rendering the entire page.

    ->When a component's state or props change, React creates a new Virtual DOM. It then compares this new Virtual DOM with the previous one using the diffing algorithm. 
     This comparison identifies the minimum set of changes required to make the real DOM match the new Virtual DOM.

    ->The diffing algorithm is optimized to work efficiently by following a few assumptions:
        1. If two elements have different types (for example, <div> changes to <span>), React removes the old element and creates a new one.
        2. If two elements are of the same type, React updates only the changed attributes instead of replacing the entire element.
        3. When rendering lists, React uses the key prop to identify which items have been added, removed, or reordered. 
          Stable and unique keys help React update lists efficiently and avoid unnecessary re-renders.

    This entire process makes React fast because it minimizes direct manipulation of the real DOM, which is relatively expensive.

Question 3: Explain React Server Components and their advantages. ?
Answer:
    ->React Server Components (RSC) are components that are rendered on the server instead of the browser. 
      They allow you to fetch data and generate UI on the server, sending only the rendered result to the client. 
      Unlike Client Components, Server Components do not include client-side JavaScript by default, making applications faster and more efficient.

    ->Server Components are commonly used in frameworks like Next.js (App Router).

    ->Advantages:
        Smaller JavaScript bundle: Since Server Components don't send their JavaScript to the browser, the client downloads less code.
        Better performance: Less JavaScript means faster page loads and quicker rendering.
        Direct data fetching: Server Components can fetch data directly from databases or APIs without exposing secrets to the client.
        Improved SEO: The HTML is generated on the server, making it easier for search engines to crawl.
        Enhanced security: Sensitive logic, API keys, and database queries remain on the server.
    
    ->Example:
        // Server Component
        export default async function Products() {
          const products = await getProducts();

          return (
            <div>
              {products.map(product => (
                <p key={product.id}>{product.name}</p>
              ))}
            </div>
          );
        }

        Here, getProducts() runs on the server, and only the rendered HTML is sent to the browser.

    ->Difference between Server Components and Client Components:
      | Server Components                               | Client Components                                    |
      | ----------------------------------------------- | ---------------------------------------------------- |
      | Render on the server                            | Render in the browser                                |
      | Can fetch data directly                         | Fetch data using APIs                                |
      | Don't send JavaScript to the client by default  | JavaScript is downloaded and executed in the browser |
      | Cannot use hooks like `useState` or `useEffect` | Can use React hooks and browser APIs                 |
      | Best for static content and data fetching       | Best for interactive UI                              |

    ->Interview Summary:
        "React Server Components are rendered on the server, allowing data fetching and UI generation before sending the result to the browser. 
         They reduce the client-side JavaScript bundle, improve performance and SEO, and keep sensitive logic on the server. 
         However, they cannot use client-side hooks like useState or useEffect, so interactive features are implemented using Client Components."

Question 4: How would you implement infinite scrolling efficiently?
Answer:
    I would use the Intersection Observer API to detect when the user reaches the bottom of the list,
    then fetch the next page of data from the server. 
    I would maintain pagination state, display a loading indicator, and stop requesting data when no more results are available. 
    For large datasets, I would also use list virtualization libraries like react-window or react-virtualized so that only the 
    visible items are rendered, improving performance and reducing memory usage."

Question 5: What are Core Web Vitals?
Answer:
      Core Web Vitals are a set of performance metrics defined by Google to measure the real-world user experience of a website.

      They focus on three key aspects:
          Loading Performance – How quickly the main content loads.
          Interactivity – How quickly the page responds to user interactions.
          Visual Stability – Whether the page layout remains stable while loading.

      The three Core Web Vitals are:

        1. Largest Contentful Paint (LCP):
              Measures loading performance.
              It records how long it takes for the largest visible content (such as an image or heading) to appear.
              Good score: 2.5 seconds or less

        2. Interaction to Next Paint (INP)
              Measures responsiveness.
              It records how quickly the page responds after a user interaction, such as a click or key press.
              Good score: 200 ms or less

        3. Cumulative Layout Shift (CLS)
              Measures visual stability.
              It tracks unexpected layout shifts while the page is loading.
              Good score: 0.1 or less

        Interview Summary:
            "Core Web Vitals are Google's key metrics for measuring user experience. They include Largest Contentful Paint (LCP) for loading speed,
             Interaction to Next Paint (INP) for responsiveness, and Cumulative Layout Shift (CLS) for visual stability. Optimizing these metrics
             improves user experience, SEO, and overall website performance."

Question 6: What techniques would you use to improve Core Web Vitals?
Answer:
  To improve Core Web Vitals, I would optimize Largest Contentful Paint (LCP) by compressing images, using lazy loading for non-critical images,
                                  preloading important assets, and reducing JavaScript bundle size. 
  To improve Interaction to Next Paint (INP), I would minimize long-running JavaScript tasks, debounce or throttle event handlers, and 
                                        avoid unnecessary React re-renders using techniques like React.memo, useMemo, and useCallback. 
  To reduce Cumulative Layout Shift (CLS), I would always specify image and video dimensions, reserve space for dynamic content, 
           and avoid inserting content above existing elements. 
           
  I would also use code splitting, caching, and a CDN to improve overall page performance."

  Short Points to Remember:
    LCP: Optimize images, preload critical assets, reduce bundle size.
    INP: Reduce JavaScript execution time, optimize event handlers, avoid unnecessary re-renders.
    CLS: Set width/height for images, reserve space for dynamic content, avoid layout shifts.
    Overall: Code splitting, lazy loading, caching, CDN, compression.

Question 7:How do you manage state in large React applications?
Answer:
  In large React applications, I manage state based on its scope and complexity. I use local state (useState) for component-specific data,
  Context API for simple global state like themes or authentication, and Redux Toolkit for complex shared state such as user data, shopping carts, 
  or application-wide settings. 
  For server state, I prefer libraries like TanStack Query to handle data fetching, caching, and synchronization.
   I also keep the state normalized, avoid unnecessary global state, and split it into feature-based slices to make the application scalable and maintainable."

  State Management Strategy:
    Local State (useState) → Component-specific data
    Context API → Authentication, theme, language
    Redux Toolkit → Complex global state
    TanStack Query → API data, caching, synchronization
