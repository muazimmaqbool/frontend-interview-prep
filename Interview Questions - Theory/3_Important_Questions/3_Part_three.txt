Question 1:  Differences between framework-level event systems and native browser events ?
Answer:
    Native browser events are the events provided directly by the browser through the DOM API, such as addEventListener(). 
    Frameworks like React provide their own event system (Synthetic Events) that wraps native events to offer a consistent API across different browsers and optimize event handling. 
    In React, events are delegated internally, reducing the number of event listeners attached to the DOM.

    Comparison:
        | Native Browser Events                 | Framework-Level Events (React)                      |
        | ------------------------------------- | --------------------------------------------------- |
        | Uses `addEventListener()`             | Uses React's event handlers like `onClick`          |
        | Browser-specific behavior may vary    | Provides a consistent API across browsers           |
        | Attached directly to DOM elements     | Managed by React's event system                     |
        | You manually add and remove listeners | React manages listeners automatically               |
        | Uses native `Event` object            | Uses `SyntheticEvent` (wrapper around native event) |

    Example:
        
        ->Native Browser Event:

            const button = document.getElementById("btn");
            button.addEventListener("click", () => {
                console.log("Clicked");
            });

        ->React Event:
            function App() {
            return (
                <button onClick={() => console.log("Clicked")}>
                    Click Me
                </button>
            );
            }

        ->Interview Summary:
            Native browser events use the DOM's addEventListener() API and expose the browser's Event object. 
            Frameworks like React use a Synthetic Event system that wraps native events to provide consistent cross-browser behavior and centralized event handling, 
            allowing developers to work with a simpler and more predictable event API."

Question 2: Difference Between Hooks and Higher-Order Components (HOCs)?
Answer:
    Hooks and Higher-Order Components (HOCs) are both patterns for reusing logic in React, but they work differently. 
    ->A Hook is a function that lets functional components use React features like state and lifecycle methods. 
    ->A Higher-Order Component (HOC) is a function that takes a component as input and returns a new enhanced component with additional functionality.

    ->Comparison:
        | Hooks                                            | HOCs                                   |
        | ------------------------------------------------ | -------------------------------------- |
        | Introduced in React 16.8                         | Used before Hooks for code reuse       |
        | Used inside functional components                | Wrap existing components               |
        | Share logic without changing component hierarchy | Create a new enhanced component        |
        | Simpler and easier to read                       | Can lead to "wrapper hell" if overused |
        | Preferred in modern React                        | Less common in new applications        |

    ->Hook Example (Built-in):
        useState()
        useEffect()
        useContext()
        useReducer()
        useMemo()
        useCallback()
        useRef()

    ->Built-in HOC Example:
        React.memo()
         const MyComponent = React.memo(function MyComponent(props) {
                                return <h1>{props.name}</h1>;
                            });

Question 3: How React works internally?
Answer: 
    React works by creating a Virtual DOM, which is a lightweight copy of the real DOM. 
    Whenever the state or props of a component change, React re-renders that component to create a new Virtual DOM. 
    It then compares the new Virtual DOM with the previous one using the reconciliation (diffing) algorithm. 
    React identifies the minimum changes required and updates only those parts of the real DOM, making UI updates efficient and improving performance

    ->React Rendering Flow:
        Component Render
                ↓
             JSX
                ↓
        React.createElement()
                ↓
        Virtual DOM
                ↓
        State/Props Change
                ↓
        New Virtual DOM
                ↓
        Reconciliation (Diffing)
                ↓
        Minimal Real DOM Updates
                ↓
        Browser Paint
    ->Key Concepts:
        JSX → Converted to React.createElement()
        Virtual DOM → Lightweight copy of the UI
        Reconciliation → Compares old and new Virtual DOM
        Diffing Algorithm → Finds minimal changes
        React Fiber → Prioritizes and schedules updates
        Real DOM Update → Only changed elements are updated
                        
    ->Interview Summary (Best Answer):
        "Internally, React converts JSX into JavaScript objects using React.createElement() and builds a Virtual DOM.
         When state or props change, React creates a new Virtual DOM and compares it with the previous one using the reconciliation (diffing) algorithm. 
         It calculates the minimum set of changes and updates only those parts of the real DOM. React's Fiber architecture further improves 
         performance by prioritizing updates and allowing rendering work to be paused and resumed, keeping the application responsive."

Question 4: Does class declaration get hoisted?
Answer:
    Yes, class declarations are hoisted, but they cannot be accessed before they are declared because they are in the Temporal Dead Zone (TDZ).

    ->Class declarations are hoisted, similar to let and const, but they are not initialized during hoisting. 
      They remain in the Temporal Dead Zone until their declaration is reached. 
      If you try to access a class before it's declared, JavaScript throws a ReferenceError.

    ->Example:
        const obj = new Person(); // ❌ ReferenceError

        class Person {
            constructor(name) {
                this.name = name;
            }
        }
        o/p: ReferenceError: Cannot access 'Person' before initialization
