Question 1: Explain frontend caching strategies.
Answer:
    Frontend caching improves application performance by storing frequently used resources so they don't need to be fetched from the server every time. 
    Different caching strategies are used depending on the type of data. 
        ->For static assets like JavaScript, CSS, and images, I rely on browser caching with proper cache headers. 
        ->For API data, I use libraries like TanStack Query, which cache responses and automatically refetch stale data when needed. 
        I also use localStorage or sessionStorage for small client-side data and Service Workers for offline support and 
        advanced caching in Progressive Web Apps (PWAs)

    Common Frontend Caching Strategies:
    Browser Cache: Caches static assets (JS, CSS, images) using HTTP cache headers.
    API Caching: Cache API responses using TanStack Query or similar libraries.
    localStorage / sessionStorage: Store small pieces of data like user preferences or tokens (avoid storing sensitive tokens unless appropriate).
    Service Workers: Cache assets and API responses for offline access and faster loading in PWAs.
    CDN Caching: Serve static assets from geographically distributed servers to reduce latency.

    Interview Summary:
        "I use browser caching for static assets, TanStack Query for API response caching, localStorage or sessionStorage for small client-side data,
        Service Workers for offline caching, and CDNs to deliver static content quickly. 
        Choosing the right caching strategy improves performance, reduces server load, and enhances the user experience."

    SubQuestion: Do browser cache works automatically?
        ->
        Ans: Yes, browser caching works automatically, but only if the server provides the appropriate caching headers.

Question 2: How do you implement authentication in a React application?
Answer:
    To implement authentication in a React application, 
        -> I first create a login page where users submit their credentials. 
        -> The backend validates them and returns an access token or session. 
        -> I store the authentication state using Context API or Redux, and then protect private routes with a reusable ProtectedRoute component. 
        -> For security, I prefer storing tokens in HTTP-only cookies because they cannot be accessed by JavaScript, reducing the risk of XSS attacks. 
        -> On application startup, I verify whether the user is still authenticated by checking the session or validating the token. 
        -> If authentication fails or the token expires, I log the user out and redirect them to the login page.

        ->Authentication Flow:
            User logs in
                ↓
            Backend verifies credentials
                ↓
            Returns JWT or Session
                ↓
            Store auth state (Context API / Redux)
                ↓
            Protect routes using ProtectedRoute
                ↓
            Verify token/session on app load
                ↓
            Logout if token expires or is invalid

Question 3: What is the difference between Session Storage and Local Storage?
Answer:
    Both Local Storage and Session Storage are browser storage mechanisms used to store key-value pairs on the client side. The main difference is how long the data persists.

    | Local Storage                               | Session Storage                                                        |
    | ------------------------------------------- | ---------------------------------------------------------------------- |
    | Data persists until manually removed.       | Data is cleared when the browser tab or window is closed.              |
    | Shared across all tabs of the same origin.  | Unique to each browser tab.                                            |
    | Storage limit is typically around 5–10 MB.  | Storage limit is typically around 5–10 MB.                             |
    | Suitable for user preferences, themes, etc. | Suitable for temporary data like form progress or a multi-step wizard. |

    ->Example:
        1: Local Storage:
            localStorage.setItem("theme", "dark");
            const theme = localStorage.getItem("theme");
            localStorage.removeItem("theme");
        
        2. Session Storage:
            sessionStorage.setItem("username", "John");
            const username = sessionStorage.getItem("username");
            sessionStorage.removeItem("username");

    ->Interview Summary:
        The main difference is that Local Storage persists even after the browser is closed until it's manually cleared,
        while Session Storage lasts only for the lifetime of the current browser tab and is cleared when the tab is closed.  
        I use Local Storage for persistent data like user preferences and Session Storage for temporary data that's only needed during the current browsing session

    ->Interview Tip:
        Avoid storing sensitive information (such as JWT access tokens or passwords) in either Local Storage or Session Storage because they are accessible 
        through JavaScript and can be exposed in an XSS attack. For authentication, HTTP-only cookies are generally the more secure choice.

Question 4: How does addEventListener work?
Answer:
    addEventListener() attaches an event listener (callback function) to an HTML element.
    When the specified event occurs (like click, input, or keydown), the browser invokes the callback.
    
    Example:
        const button = document.getElementById("btn");

        button.addEventListener("click", () => {
            console.log("Button clicked!");
        });

        //Important removing evenListeners:
            button.removeEventListener("click", handleClick);

    Extra but important:
        -> How it works internally
            button.addEventListener("click", handleClick);

            addEventListener()
                    │
                    ▼
            Browser stores the callback in the element's event listener list
                    │
                    ▼
            User performs an action (e.g., clicks the button)
                    │
                    ▼
            Browser creates an Event object
                    │
                    ▼
            Event propagates through the DOM
            (Capturing → Target → Bubbling)
                    │
                    ▼
            Browser finds matching listeners for that event
                    │
                    ▼
            Callback function is pushed to the Call Stack
                    │
                    ▼
            Your callback executes

        ->Event travels through the DOM:
            Window
            ↓
            Document
            ↓
            html
            ↓
            body
            ↓
            div
            ↓
            button

            Capturing phase (top → target)
            Target phase
            Bubbling phase (target → top)

        ->Why use addEventListener instead of onclick?

            button.onclick = fn1;
            button.onclick = fn2; // fn1 is replaced

            With addEventListener:

            button.addEventListener("click", fn1);
            button.addEventListener("click", fn2);

            Both fn1 and fn2 will execute.

Question 5: How do you handle API rate limits gracefully on the frontend? 
Answer:
    To handle API rate limits gracefully, 
      I reduce unnecessary requests using debouncing/throttling, 
      cache responses with libraries like React Query, 
      retry failed requests with exponential backoff for 429 errors, 
      respect the Retry-After header if provided, 
      disable repeated user actions while a request is in progress, 
      and show a user-friendly message instead of continuously retrying.
